# Heimdall 🌈 — 产品需求文档（PRD）

> **1.0 Slogan**：用一张任务表，把 YouTube 口播稿创作跑通——Claude Code 写稿，OpenClaw 爬资料，Heimdall 把它们串起来。

---

## 1. 项目命名

### 1.1 名字

**Heimdall**（海姆达尔）

### 1.2 命名寓意

源自北欧神话中**彩虹桥 Bifrost 的守望者** Heimdall。他站立在连接神界（阿斯加德）与人界（米德加德）的桥头

---

## 2. 项目愿景

### 2.1 长期愿景

构建一个 Harness 编排中枢，让分布式的、异构的多个 Harness 系统像一个团队一样协同完成复杂任务。

### 2.2 1.0 收敛

**1.0 不做通用编排器**。1.0 锁定一个具体场景作为**验证载体**：

> **YouTube 口播稿创作** —— 把这个流程拆成任务表，让任务表驱动整个事件完成。

**为什么选这个场景验证 1.0**：
- ✅ 流程清晰 → 天然适合任务表驱动
- ✅ 调研阶段需要爬中文资源 → 自然触发 Harness 切换（Claude Code → OpenClaw → Claude Code）
- ✅ 最终产物明确 → 易于判断 1.0 是否真跑通

**收敛方法论**：先用具体场景验证「任务表驱动 + Harness 切换」的可行性，跑通后再抽象成通用框架。**避免过早抽象**。

## 3. 1.0 核心场景：YouTube 口播稿创作 ⭐

> 本章是 PRD 的**主线**。1.0 的所有功能需求都围绕这个场景展开。

### 3.1 场景定义

用户输入一个视频选题（如「内丹法是什么」），Heimdall 自动跑完下面 5 步任务表，最终输出一整套可发布的口播稿物料。

### 3.2 工作流（5 步任务表）

| # | 任务 ID          | 步骤           | 关键产物                        | 执行 Harness           | 依赖            |
|---|------------------|----------------|---------------------------------|------------------------|-----------------|
| 1 | `topic_init`     | **选题立项**   | 选题卡（主题/受众/时长/定位）   | Claude Code            | -               |
| 2 | `research` ⭐    | **资料调研**   | 调研笔记（数据/案例/引用）      | **OpenClaw**（爬中文资源） | `topic_init`    |
| 3 | `outline`        | **结构设计**   | 大纲（钩子 + 主体分点 + CTA）   | Claude Code            | `research`      |
| 4 | `script_writing` | **口播稿撰写** | 完整逐字稿                      | Claude Code            | `outline`       |
| 5 | `metadata`       | **配套文案**   | 标题/描述/标签/封面文案         | Claude Code            | `script_writing`|

⭐ **第 2 步是 1.0 的关键验证点**：任务表在这里把控制权从 Claude Code 切换到 OpenClaw，跑完再切回来。这条切换链路成立，Heimdall 的价值就立住了。

> **颗粒度决策（Q1）**：5 步不细分。配套文案（标题/描述/标签/封面）作为一个整体任务，由 Claude Code 在一个 prompt 中产出。

### 3.3 任务表 Schema

每个任务节点的数据结构：

```yaml
- id: research                       # 唯一 ID
  name: 资料调研                      # 任务名
  status: todo                       # todo | in_progress | done | blocked
  depends_on: [topic_init]           # 前置依赖
  handler: openclaw                  # 执行者：claude_code | openclaw | ...
  inputs:                            # 输入（引用前序任务的产物）
    topic_card:
      from: artifact_of
      task: topic_init
  outputs:                           # 输出（完成后填入）
    research_notes: TBD
  artifacts:                         # 物理产物（文件路径）
    - path: artifacts/research_notes.md
```

任务表整体是 Heimdall 的**唯一事实源（single source of truth）**：
- 它是**待办清单**（todo 还有哪些）
- 它是**全局状态**（每个任务到哪一步了）
- 它是**进度看板**（一眼看到全流程）
- 它是**协作媒介**（每个 Harness 看它来知道该干嘛）

> **⚠️ 实现已演进（v0.3）**：上面用「一个 YAML 文件」是非技术视角下对「共享状态」最朴素的表达。技术实现已确定为**有状态 HTTP 服务的「通用数据层」**——任务表不是一个文件，而是数据库里的通用 `Project / Task / Record` 三张表，通过 HTTP API 读写。

### 3.4 任务表驱动机制

Heimdall 的主循环很朴素：

```
loop:
    扫描 taskboard.yaml
    找出 status=todo 且 depends_on 全部 done 的任务
    若没有 → 流程结束（或全部 blocked → 异常退出）
    取第一个匹配任务 t：
        t.status = in_progress
        根据 t.handler，把 t + 其依赖产物，派发给对应 Harness
        等待 Harness 返回产物
        写回 t.outputs / artifacts
        t.status = done（或失败时 blocked）
    持久化 taskboard.yaml
    回到 loop
```

**幂等可恢复**：任何时刻挂掉，重启后从 taskboard.yaml 现状继续，已 done 的不重跑。

### 3.5 已拍板的关键决策

| 决策点                        | 选择           | 含义                                                             |
| ----------------------------- | -------------- | ---------------------------------------------------------------- |
| **Q1** 步骤颗粒度             | 不细分         | 5 步正好，配套文案作为一个整体任务                               |
| **Q2** 任务表生成方式         | ~~A - 预定义模板~~ → **演进**为 Agent 种子行为 | 5 步流程是「规划 Agent」读种子配置后铺出来的，**不是服务端内置模板**；服务端不区分静态/动态 |
| **Q3** 1.0 是否真切换 Harness | A - 是         | 1.0 就在调研阶段真实切到 OpenClaw，验证 Heimdall 价值                |

## 4. 需求设计

### 4.1 背景与动机

#### 4.1.1 现状痛点

在使用 Claude Code、OpenClaw、Hermes、Coze 等 Harness 系统时，发现：

- **生态割裂**：每个 Harness 有自己擅长的领域和生态资源（Claude Code 擅长代码与本地文件、OpenClaw 擅长中文互联网爬取与工作流），但它们之间**信息交互成本高**
- **任务割裂**：一个复杂任务常常需要跨生态协同（如调研阶段需要爬取某个中文站点 → Claude Code 拿不到 → 需要切到 OpenClaw 爬，再把结果回流）
- **状态割裂**：用户必须在多个 Harness 之间手动搬运上下文、复制中间产物、复述任务进度
- **协同失效**：当某个子任务在 A 系统失败、需要 B 系统兜底时，缺乏统一的调度机制

#### 4.1.2 1.0 想解决的问题（聚焦）

1.0 不解决所有问题，**只解决一个具体痛点**：

> 创作者做 YouTube 口播视频时，要手动在 Claude Code（写稿）和 OpenClaw（爬资料）之间来回切换、复制粘贴。Heimdall 把这个流程自动化。

### 4.2 用户故事（1.0 范围）

- **作为内容创作者**：我希望输入一个选题，Heimdall 自动产出可发布的口播稿物料，这样我可以专注内容本身，不用在多个工具间切换。
- **作为 Heimdall 开发者**：我希望任务表是 YAML 文件、状态机驱动，这样我可以读懂、调试、手动干预。
- **作为 Harness 接入方（未来）**：我希望只要按某个接口约定实现，我的工具就能作为 handler 加入任务表。

### 4.3 功能性需求（1.0 范围）

| 编号   | 需求                                                       | 优先级 |
| ------ | ---------------------------------------------------------- | ------ |
| **F1** | 启动项目时，从预定义模板生成 5 步任务表 `taskboard.yaml`   | P0     |
| **F2** | 任务表驱动的主循环（扫描→派发→回写→持久化）               | P0     |
| **F3** | Claude Code handler：接收任务 + 上下文，产出指定 artifacts | P0     |
| **F4** | OpenClaw handler：接收调研任务，爬取并产出调研笔记             | P0     |
| **F5** | 任务表持久化（YAML 文件）+ 幂等可恢复                      | P0     |
| **F6** | 基础日志：每步任务的状态变化、产物路径                     | P1     |
| **F7** | 异常处理：handler 失败时标记 blocked，不崩溃               | P1     |
| **F8** | 命令行界面：`heimdall start <选题>` 启动流程               | P1     |

### 4.4 非功能性需求（1.0 范围）

- **可观测性（最低限度）**：能从日志看到当前在跑哪个任务、产出到哪
- **可恢复**：进程被杀后，重启能继续
- **可调试**：任务表是纯文本 YAML，可手动编辑、查看
- **可手动干预**：用户能直接改 `taskboard.yaml` 强制重跑某步（如把 done 改回 todo）

> 性能、可扩展性、安全隔离等**不在 1.0 范围**。

### 4.5 边界与非目标（1.0 明确不做）

- ❌ Heimdall 1.0 **不自己执行**任何子任务（不爬数据、不写稿、不做推理）
- ❌ Heimdall 服务端**不内置编排逻辑**（无 DAG 引擎、无能力路由、无扇入/扇出）——这些是 Agent 行为
- ❌ 1.0 **不接入** Claude Code 和 OpenClaw 之外的 Harness（Coze / Hermes 等留到后续）
- ❌ 1.0 **不做** 多用户、权限、性能扩展

> **❎ 已删除的旧非目标「不做动态任务规划」**：在「通用数据层」架构下，动态规划只是 Agent 多调几次 `POST /tasks`，服务端无感。该非目标失效。

## 5. 待讨论问题（Open Questions）

> 1.0 主线已收敛，以下问题主要影响**实现细节**，可边做边定：

1. **运行形态**：Heimdall 是 CLI 工具（每次执行跑一遍），还是常驻本地服务？
2. **任务表存储**：YAML 文件 vs SQLite？（目前倾向 YAML，可读性优先）
3. **handler 通信协议**：
   - Claude Code：直接 stdio 调用？还是 SDK？
   - OpenClaw：调用 OpenClaw 的 API？还是用 OpenClaw 工作流？
4. **产物存放**：artifacts 是文件（建议 `artifacts/` 目录）还是嵌在 YAML 里？
5. **是否拥抱 MCP**：长期是否基于 MCP 协议？1.0 是否就要？
6. **任务失败后的兜底策略**：blocked 后是停在那等人工，还是自动尝试别的 handler？

## 6. 架构设计

已收敛为 **「通用数据层 + 机制/策略分离」** 架构：Heimdall 是一块共享黑板（有状态 HTTP 服务），只做机制（存数据 + 守最小领域规则）；所有策略（规划/分工/爬取/清洗）交给 Harness。

数据模型以 `Project / Task / Record` 三张通用表为核心，具体实现见代码仓库。

---

## 7. 技术选型

已拍板：**后端 Python + FastAPI + 关系型数据库**；前端 **React + TypeScript + Vite + Ant Design**。对象存储、MCP 等能力按版本迭代引入，不在本文档展开部署细节。

## 8. 路线图（Roadmap）

| 里程碑                       | 目标                                   | 验证标准                                         |
| ---------------------------- | -------------------------------------- | ------------------------------------------------ |
| **M0 - 单 Harness 串联**     | 任务表 + Claude Code handler 跑通 5 步 | 不切 OpenClaw，全部 Claude Code 跑完，产出物完整     |
| **M1 - 双 Harness 切换** ⭐  | 加入 OpenClaw handler，跑通真实切换        | 调研阶段真正切到 OpenClaw，产物正确回流              |
| **M2 - 健壮性**              | 异常处理 + 幂等可恢复 + 日志完善       | 任务表能手动干预，挂掉能续跑                     |
| **M3 - 抽象化**              | 抽出 handler plugin 接口               | 接入第 3 个 Harness（如 Coze）成本 < 1 天    |
| **M4 - 通用化**（v2 起步）   | 支持自定义任务表模板                   | 同一个 Heimdall 能跑别的场景（如研报、教程）     |
