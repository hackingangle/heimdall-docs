# Heimdall 调研素材收集设计（heimdall-research 技能）

- 日期：2026-07-05
- 状态：**已废弃（2026-07-06）**——决策更新：不假定 Harness 如何获取素材，信任模型自选手段，research 技能及本设计的"三通道"规定性内容随之移除。任务编排见 [素材收集任务设计](./2026-07-06-collect-task-design.md)，入库质量规则保留在 heimdall-material 技能。本文档仅作历史记录。
- 原则：简单、高可用（见 `.cursor/rules/architecture-simplicity.mdc`）；机制与策略分离——素材收集是「策略」，活在 Agent 侧，后端零改动

## 1. 背景与问题

每期节目（如「寻老和尚」）需要围绕主题收集书籍、讲稿、演讲等素材。此前用通用爬虫（OpenClaw）效果差，实际痛点四个：

1. **发现难**：不知道资料在哪，这是搜索问题不是爬取问题；
2. **内容脏**：HTML 全文混着导航/广告/无关内容；
3. **抓不到**：反爬、微信公众号、视频等来源爬虫无能为力；
4. **格式盲区**：核心素材是 PDF 书籍、视频字幕，根本不是网页。

最有价值的素材类型（按用户确认）：**视频讲经/演讲字幕、书籍 PDF、网页文章**。

## 2. 方案决策

- **放弃通用爬虫**，采用「搜索 API + LLM 阅读 Agent」为主、「已知来源定向入库」为辅。
- **第一版落为 Claude Code 技能**（`docs/skills/heimdall-research-skill.md`），零后端改动；验证有效后再考虑服务端产品化（另立设计，本期不做）。
- **可用付费 API**：博查（发现）+ Jina Reader（网页抽取），每期成本预估几元到十几元。

## 3. 三通道获取（对症下药）

| 素材类型 | 发现 | 获取 |
|---|---|---|
| 视频讲经/演讲 | 博查搜索 + YouTube 站内搜索 | `yt-dlp` 拉字幕（含自动字幕），清洗为纯文字稿 |
| 书籍 PDF | 博查搜索找下载源 | 下载后 `markitdown` 转 Markdown，整本一条入库（章节用正文标题保留） |
| 网页文章 | 搜索 API | 阅读抽取 API 抽正文得干净 Markdown |

微信公众号第一版**不硬啃**：靠博查与转载站覆盖部分，抓不到的记入调研索引留给人工。此为已知局限。

## 4. 清洗与入库规则

- **相关性判断由执行 Agent（Claude Code）承担**：每篇先速读，无关丢弃；多个转载取最完整一份。
- **素材保留原文，不过度摘要**：一本书 / 一篇演讲 / 一篇文章 / 一个视频字幕 = 一条素材（书不按章拆条）。
- **标题格式统一**：`[网页文章] 站点 - 篇名`、`[视频字幕] 频道 - 标题`、`[书籍] 书名（编著者）`。
- **正文头部附出处**：来源 URL + 获取日期，便于写稿引用溯源。
- **入库**：复用现有 `POST /api/projects/{id}/materials`（`hd_` API Token，source 自动标 `api`）。
- **每次调研额外产出一条「调研索引」素材**：收了什么、哪些来源失败、建议人工补充什么。单条失败不中断整体。

## 5. 依赖与配置

- 本地工具：`yt-dlp`、`markitdown`（一次性安装）。
- 环境变量：搜索 API Key（必需）、阅读抽取 API Key（可选）、Heimdall API Token。

## 6. 验收

用一期真实主题（「寻老和尚」）跑一遍，产出应包含：一批网页文章素材、至少一个视频字幕素材、至少一本书的全本素材，外加一条调研索引；人工抽查相关性与正文干净度。

## 7. 本期不做

服务端一键调研（产品化）、微信公众号定向抓取、通用/定向爬虫、素材自动摘要、大文件对象存储。
